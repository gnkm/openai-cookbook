# 良いドキュメントとは何か

ドキュメントは、有用な情報を他の人の頭の中に入れるものです。より良いドキュメントを書くために、以下のヒントに従ってください。

### ドキュメントを読み飛ばしやすくする

読者のほとんどは、上から下まで直線的に読むことはありません。彼らは飛び回り、どの部分が自分の問題を解決するかを評価しようとします。検索時間を短縮し、成功の確率を高めるために、ドキュメントを読み飛ばしやすくしましょう。

**コンテンツをタイトル付きのセクションに分割する。** セクションタイトルは道標として機能し、読者に注目すべきか、先に進むべきかを伝えます。

**抽象的な名詞よりも情報的な文章をタイトルに使用する。** たとえば、「結果」のようなタイトルを使用すると、読者は実際の結果が何であるかを知るために、後続のテキストに飛び込む必要があります。対照的に、「ストリーミングにより最初のトークンまでの時間が50%短縮された」というタイトルを使用すると、読者に余分な手間をかけずに、すぐに情報を提供できます。

**目次を含める。** 目次は、ハッシュマップがリンクリストよりも高速な検索を持つのと同様に、読者が情報をより速く見つけるのに役立ちます。目次には、見過ごされがちな2つ目の利点もあります：読者にドキュメントに関する手がかりを与え、読む価値があるかどうかを理解するのに役立ちます。

**段落を短く保つ。** 短い段落は読み飛ばしやすくなります。重要なポイントがある場合は、見逃される可能性を減らすために、それを独立した一文の段落にすることを検討してください。長い段落は情報を埋もれさせる可能性があります。

**段落とセクションを、独立したプレビューを提供する短いトピック文で始める。** 人々が読み飛ばすとき、彼らはセクションの最初の単語、最初の行、最初の文を不釣り合いに見ます。これらの文を、前のテキストに依存しない方法で書いてください。たとえば、最初の文「これに基づいて、より速い方法について話しましょう」を考えてみましょう。この文は、前の段落を読んでいない人には意味がありません。代わりに、独立して理解できる方法で書いてください：例えば、「ベクトルデータベースは埋め込み検索を高速化できます。」

**トピック文の冒頭にトピックワードを置く。** 読者は、段落が何についてかを知るために1語か2語だけを読む必要がある場合、最も効率的に読み飛ばすことができます。したがって、トピック文を書くときは、文の最後ではなく冒頭にトピックを置くことを優先してください。たとえば、埋め込み検索に関する長い記事の途中でベクトルデータベースについて段落を書いていると想像してください。「埋め込み検索はベクトルデータベースによって高速化できる」と書く代わりに、「ベクトルデータベースは埋め込み検索を高速化する」を優先してください。2番目の文は、段落のトピックを段落の冒頭に置くため、読み飛ばしに適しています。

**要点を前面に出す。** 最も重要な情報をドキュメントとセクションの上部に配置します。ソクラテス的な大きな盛り上がりを書かないでください。結果の前に手順を紹介しないでください。

**箇条書きと表を使用する。** 箇条書きリストと表は、ドキュメントを読み飛ばしやすくします。頻繁に使用してください。

**重要なテキストを太字にする。** 読者が見つけやすくするために、重要なテキストを太字にすることを恐れないでください。

### 上手に書く

ひどく書かれたテキストは読むのに負担がかかります。上手に書くことで、読者への負担を最小限に抑えましょう。

**文を簡潔に保つ。** 長い文を2つに分割します。副詞を削除します。不要な単語やフレーズを削除します。該当する場合は、命令形を使用します。執筆の本が教えることを実行してください。

**明確に解析できる文を書く。** たとえば、「Title sections with sentences.」という文を考えてみましょう。読者が「Title」という単語を読むとき、彼らの脳はまだ「Title」が名詞、動詞、形容詞のいずれになるかを知りません。文の残りを解析する際に追跡するのに少し脳力が必要であり、脳が意味を誤って予測した場合に引っかかりを引き起こす可能性があります。たとえ長くても、より簡単に解析できる文（例：「Write section titles as sentences」）を優先してください。同様に、「Bicycle clearance exercise notice」のような名詞句は、解析に余分な労力がかかる可能性があるため、避けてください。

**左分岐文を避ける。** 言語ツリーは、文中で単語がどのように関連しているかを示します。左分岐ツリーは、幅優先探索と深さ優先探索に似て、右分岐文よりも読者がより多くのことを記憶に保持する必要があります。左分岐文の例は、「You need flour, eggs, milk, butter and a dash of salt to make pancakes.」です。この文では、「you need」が何に接続するかは、文の最後に到達するまでわかりません。読みやすい右分岐バージョンは、「To make pancakes, you need flour, eggs, milk, butter, and a dash of salt.」です。読者が単語をしばらく保持しなければならない文に注意し、言い換えることができるかどうかを確認してください。

**指示代名詞（例：「this」）を避ける、特に文をまたいで。** たとえば、「Building on our discussion of the previous topic, now let's discuss function calling」と言う代わりに、「Building on message formatting, now let's discuss function calling」を試してください。2番目の文は、読者に前のトピックを思い出す負担をかけないため、理解しやすくなります。指示代名詞を完全に削除する機会を探してください：例えば、「Now let's discuss function calling.」

**一貫性を保つ。** 人間の脳は驚くべきパターンマッチャーです。不一致は読者を苛立たせたり、気を散らしたりします。どこでもタイトルケースを使用する場合は、タイトルケースを使用します。どこでも末尾のカンマを使用する場合は、末尾のカンマを使用します。Cookbook のすべてのノートブックがアンダースコアと文の大文字小文字で名前が付けられている場合は、アンダースコアと文の大文字小文字を使用します。読者に「ふーん、それは変だ」と思わせるようなことをしないでください。内容に集中できるようにし、不一致に集中させないようにしてください。

**読者が何を考えているか、何をすべきかを伝えない。** 「Now you probably want to understand how to call a function」や「Next, you'll need to learn to call a function.」のような文を避けてください。どちらの例も読者の心の状態を推定しており、それが彼らを苛立たせたり、私たちの信頼性を損なったりする可能性があります。読者の状態を推定しないフレーズを使用してください。例：「To call a function, …」

### 広く役立つ

人々はさまざまなレベルの知識、言語能力、忍耐力を持ってドキュメントに来ます。経験豊富な開発者をターゲットにしている場合でも、すべての人に役立つドキュメントを書くように努めるべきです。

**シンプルに書く。** 必要だと思うよりもシンプルに物事を説明してください。多くの読者は英語を第一言語として話さないかもしれません。多くの読者は技術用語について本当に混乱しており、英語の文を解析するために費やす余分な脳力がほとんどないかもしれません。シンプルに書いてください。（ただし、過度に単純化しないでください。）

**略語を避ける。** 物事を書き出してください。専門家へのコストは低く、初心者への利益は高いです。IF の代わりに、instruction following と書いてください。RAG の代わりに、retrieval-augmented generation（または私の好みの用語：search-ask procedure）と書いてください。

**潜在的な問題への解決策を提供する。** 読者の95%が Python パッケージのインストール方法や環境変数の保存方法を知っている場合でも、それを積極的に説明する価値があります。説明を含めることは専門家にとってコストがかかりません—彼らはそれをすぐに読み飛ばすことができます。しかし、説明を除外することは初心者にとってコストがかかります—彼らは立ち往生したり、私たちを放棄したりする可能性があります。専門の JavaScript エンジニアや C++ エンジニアでさえ、Python では初心者である可能性があることを忘れないでください。説明しすぎる方向に誤ってください。

**具体的で正確な用語を優先する。** 専門用語は悪いです。自分自身ではなく、その分野に新しい人々のためにドキュメントを最適化してください。たとえば、「prompt」と書く代わりに、「input」と書いてください。または「context limit」と書く代わりに、「max token limit」と書いてください。後者の用語はより自明であり、ベースモデル時代に開発された専門用語よりもおそらく優れています。

**コード例を一般的でエクスポート可能に保つ。** コードデモンストレーションでは、依存関係を最小限に抑えるようにしてください。ユーザーに追加のライブラリをインストールさせないでください。異なるページやセクション間を行ったり来たりする必要がないようにしてください。例をシンプルで自己完結したものにしてください。

**価値によってトピックに優先順位を付ける。** 一般的な問題（例：トークンの数え方）をカバーするドキュメントは、まれな問題（例：絵文字データベースの最適化方法）をカバーするドキュメントよりも桁違いに価値があります。それに応じて優先順位を付けてください。

**悪い習慣を教えない。** API キーをコードに保存すべきでない場合は、API キーをコードに保存する例を決して共有しないでください。

**広い導入でトピックを紹介する。** たとえば、優れたレコメンダーをプログラムする方法を説明する場合は、YouTube 動画から Amazon のアイテム、Wikipedia まで、Web 全体でレコメンデーションが広く普及していることを簡単に言及することから始めることを検討してください。狭いトピックを広い導入で基礎づけることは、不確実な領域に飛び込む前に、人々がより安心感を感じるのに役立ちます。そして、テキストがよく書かれている場合、すでに知っている人もそれを楽しむかもしれません。

### 正当な理由がある場合はこれらのルールを破る

最終的には、最善だと思うことを行ってください。ドキュメントは共感の実践です。読者の立場に立って、彼らを最も助けると思うことを行ってください。
